# 読書管理アプリへの認証・ID管理システム導入：要件定義

個人開発でAWS上に読書管理アプリ（CloudFront + S3 + API Gateway + Lambda + DynamoDB のサーバレス構成）を作っていたのですが、認証とID管理が入っていない状態だったため、ここを補強することにしました。本記事はそのシリーズ第1回で、要件定義のフェーズで考えたことをまとめたものです。

## はじめに

既存の読書管理アプリは認証なしで動いており、誰でも本を登録できる状態でした。セキュリティリスクを考え、認証とID管理の仕組みを追加することにしました。ポートフォリオとして見ていただく際は、操作権限を絞ったテストユーザを1つ用意して、そのアカウントで動作確認してもらう想定です。

要件定義から基本設計、構築・検証まで一通り進める想定で、本記事はその最初のフェーズにあたります。Cognitoや監査ログまわりの設計判断を、なぜそう決めたかも含めて書いています。

## 結論

要件定義のアウトプットは、以下の方針に落ち着きました。

- スコープはMVPに絞り、管理者向け機能7つ・非機能要件10項目に限定する
- 利用者情報のマスタはAmazon Cognito User Poolに集約し、独自のユーザDBは持たない
- 監査ログは管理者操作のビジネス文脈をDynamoDBに、AWS APIレベルのイベントをCloudTrailに分担させる
- User Poolは管理者・利用者で共通の1つにまとめ、Cognito Groupで権限を分ける

ここで決めたのは「最低限なにを作るか」と「なぜそう作るか」までで、API設計やテーブル設計などの具体は次の基本設計フェーズに回します。

## 既存ポートフォリオと今回の追加スコープ

### 既存ポートフォリオの構成

既存の読書管理アプリは、フロントを CloudFront + S3、APIを API Gateway + Lambda、データを DynamoDB に置いた構成です。インフラはCDKで管理しています。利用者は本を登録・一覧・編集・削除できますが、認証が入っておらず、登録データもユーザーごとに分かれていません。

### 今回追加するもの

認証とID管理（管理者が利用者を管理する仕組み）を追加します。具体的には、利用者がアプリにアクセスする際にログインを求める仕組みと、管理者が利用者を作成・編集・無効化したり、パスワードリセットや監査ログ確認を行ったりする管理者向け機能です。

あわせて、利用者は自分が登録した本だけを参照・編集できるよう、ユーザーIDをキーにした認可制御も既存APIに加えます。

MVPでは管理機能を7つに絞り、管理者向けのWeb画面（管理画面）として最低限の構成で作る想定です。要件定義の役目は「どこまで作って、どこを切るか」スコープを整理しておくことになります。

## システム構成

中心になるのはAmazon Cognito User Poolで、利用者情報のマスタはここに集約します。管理者操作の監査ログだけはDynamoDBに別テーブルを作って書き込み、認証イベントなどAWS APIレベルのログはCloudTrailに任せます。

既存のAPI Gateway + Lambdaには認可機能を足し、Cognitoが発行するJWTトークンを検証してから業務ロジックに進む形にします。利用者は最初にCognitoのHosted UIにリダイレクトされ、ログイン後にトークンを持ってアプリへ戻ってくる流れです。

## 設計判断のポイント

### 判断1: MVPに絞り込んだ理由

機能を「あれもこれも」と盛り込んでいくときりがありません。まずは小さく作って動かし、運用の中で出てきた課題や要望に合わせて都度修正・拡張していく方が、実態に合った形に育てられます。それを実現するには、最初の段階で拡張性を持たせた構成を組んでおくことが鍵だと判断しました。

その方針のもと、MVPは管理者向け7機能に絞っています。Cognito Group や監査ログ用テーブルなど、後から機能を追加する際に組み込みやすい受け皿を残してあり、MVPで切った機能も要件定義書に「MVP外」として明記して継ぎ足せる余地を残しています。

### 判断2: Cognitoに利用者情報を集約した理由

利用者情報をどこに置くかで、当初は3案を比較していました。

- 案A: Cognitoとは別にユーザDBを持つ
- 案B: Cognito User Poolに集約する
- 案C: ハイブリッド（Cognitoに認証情報、別DBに業務属性）

最終的に案Bを選びました。MVPで管理する属性は表示名・メール・有効フラグ程度で、Cognitoの標準属性とカスタム属性でまかなえます。別DBを持つと、ユーザー情報を更新するたびにCognitoと別DBの両方に書き込む形になり、片方だけ成功してデータが食い違うリスクが出ます。それを防ぐためのリトライや不整合検知の仕組みを作り込む必要があり、MVP段階で抱え込む手数としては大きすぎると判断しました。

エンタープライズ用途では認証側にLDAPサーバ、ID管理側にMongoDBを置くような構成も珍しくないのですが、個人開発でそこまで分ける理由がありませんでした。

### 判断3: 監査ログをDynamoDBとCloudTrailで分担した理由

監査ログには2つの性格があります。「誰がどのAPIを叩いたか」というインフラ寄りの記録と、「誰が誰の何の属性をなぜ変えたか」というビジネス寄りの記録です。

前者はCloudTrailが自動で記録してくれるので、これに任せます。後者はCloudTrailのログから組み立てるのが現実的でないため、DynamoDBに専用テーブルを用意し、操作前後の差分（before/after）を入れ子のMap型で持たせる設計にしました。

DynamoDBレコードのイメージは以下のようなものです。

```json
{
  "logId": "uuid-abc123",
  "timestamp": "2026-04-30T10:23:45Z",
  "operatorId": "admin-001",
  "operationType": "UPDATE_USER",
  "targetUserId": "user-789",
  "changes": {
    "displayName": { "before": "佐藤はなこ", "after": "佐藤花子" },
    "email": { "before": "old@example.com", "after": "new@example.com" }
  },
  "reason": "本人申請による表記揺れ修正",
  "result": "SUCCESS"
}
```

DynamoDBを選んだ理由は、書き込みパターンが追記中心で、読み出しも特定ユーザの履歴をたどる程度の単純なクエリで済むためです。S3 + Athenaも候補でしたが、MVPの規模では運用が重くなりすぎると判断しました。

### 判断4: User Poolを管理者・利用者で共通化した理由

管理者用と利用者用でUser Poolを分ける案もありましたが、共通の1プールにまとめてCognito Groupで権限を分ける方式にしました。

分けると認証フローが2系統になり、後から統合する手間も増えます。共通化しておけば、将来「利用者の中から特定の人を管理者に昇格させる」といった操作もグループ付け替えだけで済みます。MVPの段階で分ける必要性がなく、必要になったタイミングで分けるコストもそれほど大きくないため、この方向に倒しました。

## 完成した要件定義書

要件定義書の本体はGitHubリポジトリの`auth-id-management/`配下に置いてあります。

- [認証・ID管理 要件定義書（GitHub）](https://github.com/wanimakokokara/reading-management-api/auth-id-management/01-requirements.md)

サマリだけ抜き出すと以下のとおりです。

| 項目 | 内容 |
| --- | --- |
| 機能要件 | 管理者ログイン / 利用者一覧 / 作成 / 編集 / 有効・無効化 / パスワードリセット / 監査ログ閲覧 |
| 非機能要件 | 認証強度（MFA）、可用性、ログ保持期間ほか計10項目 |
| 利用者マスタ | Cognito User Pool（独自DBなし） |
| 監査ログ | DynamoDB（管理者操作） + CloudTrail（AWS API） |
| User Pool構成 | 単一プール + Cognito Group |
