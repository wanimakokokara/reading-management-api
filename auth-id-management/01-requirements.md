# 読書管理アプリへの認証・ID管理システム導入 要件定義書（MVP版）

## 概要

既存の読書管理アプリ（ポートフォリオ）は認証なしで誰でも操作できる状態であり、セキュリティ面の課題を解消するため、認証基盤と ID管理システムを導入する。

- 認証基盤: Amazon Cognito を新規導入し、エンドユーザーのログイン機能を提供する
- ID管理システム: 管理者がエンドユーザーの情報編集・パスワードリセット等を行う管理画面

本書は ID管理システムの最小完成版（MVP）の要件を定義する。エンドユーザー向け認証機能は Cognito の標準機能で完結するため、本書では認証ポリシー（MFA 等）のみ記載する。フル版（拡張機能を含む）は `id-management-requirements-full.md` を参照。

### MVP の方針

- まずは最小完成版を作って動かす
- 凝った機能は後追い拡張で対応する
- ロールは管理者単一（権限分離は将来対応）
- Cognito の標準機能で済む箇所は極力 Cognito に任せる

### スコープ

- 対象: サイト管理者向けの ID管理システム
- 対象外（Cognito 標準機能に委ねる）: エンドユーザーのログイン・パスワード変更・MFA 登録等

### システム前提

- 既存システム: 読書管理アプリ（CloudFront + S3 + API Gateway + Lambda + DynamoDB、CDK 管理）
- 認証基盤: 本プロジェクトで Amazon Cognito を新規導入
- 利用者規模: エンドユーザー数人〜数十人、管理者1〜数名
- 想定将来規模: エンドユーザー数百人、管理者10名以下
- 運用体制: 自分1人（将来的に数名）
- 月額予算: 数千円程度

### 認証ポリシー

- 全ユーザーに MFA 必須
- 認証要素: TOTP（Google Authenticator または Microsoft Authenticator）
- MFA リセットは MVP では AWS コンソールから直接操作する（管理画面の機能としては実装しない）

### Cognito User Pool 構成

- MVP では **エンドユーザーと管理者を同一の User Pool に配置**し、Cognito Group により管理者を識別する
- 全ユーザーに MFA 必須というポリシーが共通であり、プールを分けるメリットが小さいため
- 将来、管理者にだけ厳しいパスワードポリシーや IP 制限を適用したい場合はプール分離を再検討する

### MVP 外（将来拡張）

以下は MVP では実装しない。フル版の要件として `id-management-requirements-full.md` に残してある。

- ユーザー詳細表示（一覧と編集画面で代替）
- ユーザーアカウント削除(無効化で代替)
- パスワード強制変更要求（リセットで代替）
- アカウント自動ロック／アンロック（Cognito 標準機能に任せる）
- ロール／権限の付与・剥奪（MVP は管理者単一ロール）
- MFA リセット（管理画面機能としては未実装、AWS コンソールで対応）
- 管理者プロファイル参照（単一管理者なら不要）

## 機能要件

### 機能1: 管理者ログイン／ログアウト

- 概要: 管理者が ID管理システムにログイン／ログアウトする。MFA チャレンジが必須。ログイン処理自体は Cognito 標準機能を使う
- アクター: 管理者
- トリガー: 管理画面のログインボタン（Cognito ログイン画面へリダイレクト）／ログアウトボタン
- 入力（ログイン、Cognito 側で受け付ける）:
  - メールアドレス（必須）
  - パスワード（必須）
  - MFA コード（必須、TOTP 6桁）
- 処理内容（ログイン）:
  1. 管理画面が Cognito のログイン画面へリダイレクト
  2. Cognito がメール・パスワード・MFA を検証
  3. 成功なら JWT トークンを発行し、管理画面へリダイレクト
  4. 初回ログインで MFA 未登録なら、Cognito の MFA 初回登録フロー（QR コード表示・TOTP コード検証）が走る
- 処理内容（ログアウト）:
  1. 管理画面でセッショントークン破棄
  2. Cognito ログアウトエンドポイントを呼び出してセッション無効化
- 出力（ログイン）: 管理画面トップへ遷移、または初回ログイン時は Cognito の MFA 登録画面へ
- 出力（ログアウト）: ログイン画面へ遷移
- エラー条件（Cognito が処理）:
  - 認証エラー: メール・パスワード不一致、MFA コード不正
  - アカウント無効化時: ログイン拒否
- 関連する非機能要件:
  - 性能: ログイン3秒以内
  - セキュリティ: MFA 必須、QR コードは初回1回限り表示
  - ログ: ログイン成功・失敗は CloudTrail に自動記録される（独自実装不要）

### 機能2: ユーザー一覧表示

- 概要: 管理者が登録ユーザーの一覧を表示する。Cognito ListUsers API を呼び出して結果を表示する
- アクター: 管理者
- トリガー: 管理画面の「ユーザー管理」メニューを選択
- 入力:
  - 検索キーワード（任意、メール・表示名で前方一致検索。Cognito の検索仕様に合わせて部分一致は非対応）
  - 1ページあたりの件数（任意、デフォルト20）
  - ページネーショントークン（任意、Cognito から返される）
- 処理内容:
  1. 認証チェック
  2. Cognito ListUsers API を呼び出し（フィルタ条件・件数・トークンを渡す）
  3. 監査ログ DynamoDB に「ユーザー一覧閲覧」を記録
  4. 結果を返す
- 出力: ユーザー一覧テーブル（メール・表示名・ステータス・登録日・最終ログイン日 等）+ 次ページトークン
- エラー条件:
  - Cognito API エラー: 一時的に利用不可時、再試行を案内
- 関連する非機能要件:
  - 性能: 検索結果表示まで3秒以内
  - ログ: 閲覧操作も DynamoDB に記録

### 機能3: ユーザー新規作成

- 概要: 管理者が新規ユーザーを Cognito に登録する
- アクター: 管理者
- トリガー: 管理画面の「ユーザー新規作成」ボタン
- 入力:
  - メールアドレス（必須、Cognito 内で一意）
  - 表示名（必須）
  - 招待メール送信フラグ（任意、デフォルト ON）
- 処理内容:
  1. 認証チェック
  2. 入力バリデーション（メール形式、必須項目）
  3. Cognito AdminCreateUser API でユーザー作成（属性も同時に設定、Cognito 機能で招待メール送信）
  4. 招待メール経由の初回ログイン時に MFA 登録が強制される（Cognito User Pool の MFA 必須設定による）
  5. 監査ログ DynamoDB に「ユーザー作成」を記録
  6. 結果を返す
- 出力: 作成完了メッセージ + 一覧画面へ戻る
- エラー条件:
  - 入力エラー: メール形式不正・必須項目欠落
  - メール重複: Cognito が `UsernameExistsException` を返したらエラー表示
  - Cognito API エラー: そのままエラーを返す（独立 DB がないのでロールバック不要）
- 関連する非機能要件:
  - 性能: 登録完了まで5秒以内
  - セキュリティ: 招待メールには有効期限付きトークン（Cognito 標準）

### 機能4: ユーザー情報編集

- 概要: 管理者が登録済みユーザーの属性を Cognito 上で編集する
- アクター: 管理者
- トリガー: ユーザー一覧画面の「編集」ボタン
- 入力:
  - ユーザーID（Cognito のユーザー識別子、必須）
  - メールアドレス（任意、変更時のみ）
  - 表示名（任意）
- 処理内容:
  1. 認証チェック
  2. 入力バリデーション（メール形式）
  3. Cognito AdminGetUser API で変更前の属性値を取得（差分記録のため）
  4. Cognito AdminUpdateUserAttributes API で属性を更新
  5. 監査ログ DynamoDB に「ユーザー情報編集」を変更前後の差分付きで記録
  6. メール変更時は Cognito 機能で本人に通知メール送信
  7. 結果を返す
- 出力: 編集完了メッセージ + 一覧画面へ戻る
- エラー条件:
  - 入力エラー: メール形式不正
  - 重複: メール変更先が既存ユーザーと衝突した場合、Cognito がエラーを返す
  - Cognito API エラー: そのままエラーを返す
- 関連する非機能要件:
  - 性能: 更新完了まで3秒以内
  - ログ: 変更前後の値を記録

### 機能5: ユーザーアカウント有効化／無効化

- 概要: 管理者が Cognito 上のユーザーアカウント状態を切り替える
- アクター: 管理者
- トリガー: ユーザー一覧画面の「無効化」「有効化」ボタン
- 入力:
  - ユーザーID（必須）
  - 状態切り替え（有効化 or 無効化）
- 処理内容:
  1. 認証チェック
  2. Cognito AdminEnableUser または AdminDisableUser API を呼び出し
  3. 監査ログ DynamoDB に「アカウント有効化／無効化」を記録
  4. 結果を返す
- 出力: 切り替え完了メッセージ + 一覧画面へ戻る
- エラー条件:
  - Cognito API エラー: そのままエラーを返す
- 関連する非機能要件:
  - 性能: 状態切り替え3秒以内
  - ログ: 状態切り替えを記録

### 機能6: パスワードリセット（管理者主導）

- 概要: 管理者が特定ユーザーのパスワードをリセットする。Cognito のパスワードリセット機能を呼び出す
- アクター: 管理者
- トリガー: ユーザー一覧画面の「パスワードリセット」ボタン
- 入力:
  - ユーザーID（必須）
- 処理内容:
  1. 認証チェック
  2. Cognito AdminResetUserPassword API を呼び出す。Cognito から本人に **確認コード付きの通知メール** が送信される。ユーザーは確認コードと新パスワードを入力してリセットを完了する
  3. 監査ログ DynamoDB に「パスワードリセット」を記録
  4. 結果を返す
- 出力: リセット完了メッセージ
- エラー条件:
  - Cognito API エラー: そのままエラーを返す
- 関連する非機能要件:
  - 性能: リセット処理3秒以内
  - ログ: パスワードリセット操作を記録

### 機能7: 監査ログ閲覧

- 概要: 管理者が DynamoDB に蓄積された監査ログを参照する。管理者操作の履歴（誰が何をしたか）を確認する
- アクター: 管理者
- トリガー: 管理画面の「監査ログ」メニュー
- 入力:
  - 期間フィルタ（任意、開始日・終了日）
  - 操作種別フィルタ（任意）
  - 1ページあたり件数
- 処理内容:
  1. 認証チェック
  2. DynamoDB から条件に合う監査ログを取得（パーティションキー・GSI を活用）
  3. 監査ログ DynamoDB に「監査ログ閲覧」を記録
  4. 結果を返す
- 出力: 監査ログ一覧（時刻、操作者、対象、操作種別、変更前後の差分、結果）
- エラー条件:
  - 入力エラー: 期間指定が不正（終了日 < 開始日）
- 関連する非機能要件:
  - 性能: 検索結果表示まで5秒以内
  - データ保持: 監査ログは長期保管、削除不可
- 補足: 認証関連操作（ログイン・パスワード変更等）は本機能には含めない。それらは CloudTrail で確認する

## システム構成

ID管理システムを構成する要素と役割を以下に定義する。

### 構成要素

- 認証基盤（Cognito）: ユーザー情報（メール・パスワード・MFA 設定・属性）の唯一の保管場所。エンドユーザーのログイン画面・パスワード変更フロー・MFA 登録フローはすべて Cognito 標準機能で完結する。**エンドユーザーと管理者は同一 User Pool に配置し、Cognito Group で管理者を識別する**
- 管理画面（CloudFront + S3）: 管理者が使う React 製のシングルページアプリケーション
- 管理 API（API Gateway + Lambda）: 管理画面からのリクエストを受け、Cognito Admin API を呼び出す。同時に監査ログを書き込む
- 監査ログ DB（DynamoDB）: 管理者の操作履歴を保存。管理画面の機能7 から検索・閲覧する

### データの所在

- ユーザー情報: Cognito にのみ保存。ID管理システム側に独自のユーザー DB は持たない
- 監査ログ: DynamoDB にのみ保存。Cognito 側には連携しない

### 監査ログの分担

- 管理者の管理操作（ユーザー作成・編集・状態切り替え・パスワードリセット・一覧閲覧 等）: Lambda から DynamoDB に書き込む
- 管理者・エンドユーザーの認証関連操作（ログイン・ログアウト・パスワード変更・MFA 操作）: Cognito の標準動作で CloudTrail に自動記録される。独自の記録処理は実装しない

## 非機能要件

### 1. セキュリティ要件

目的: 個人情報を保護し、不正アクセスから ID管理システムを守る

要件:
- 通信は全て TLS 1.2 以上で暗号化（HTTP は不可）
- ユーザー情報・監査ログは保存時に暗号化
- 全ユーザーに MFA を必須とする（TOTP）
- パスワードはハッシュ化保存（平文を持たない）
- 画面・ログにパスワード・MFA 秘密鍵・セッショントークンを出力しない
- セッションは一定時間で自動タイムアウト（具体値は基本設計で決定）

受け入れ基準:
- 全エンドポイントで HTTPS 強制リダイレクトされる
- MFA 未設定で管理画面にアクセスできない

### 2. 認証・認可要件

目的: 管理画面の操作を認証済みかつ管理者権限を持つユーザーのみに限定する。エンドユーザー側は自分が登録した本のデータのみアクセス可能とする

要件:
- MVP は管理者単一ロール（権限分離は将来対応）
- 全機能で認証チェックを実施し、バイパスできない構造とする
- 認証チェック後、Cognito Group により管理者ロールであることを確認する（MVP は単一ロールのため認証＝認可だが、将来のロール分離拡張に備えて認可ステップを独立して実装する）
- 認証・認可違反時は処理を即時停止し、監査ログに記録
- エンドユーザーが既存読書管理アプリの API を呼ぶ際、JWT トークンから取得したユーザー ID をキーに自身の登録データのみ参照・更新できるよう認可制御する（他ユーザーのデータには一切アクセスさせない）

受け入れ基準:
- 各機能で認証・認可チェックの実装が一貫している
- 認証・認可違反のログが記録される
- エンドユーザーが他ユーザーの本データにアクセスできないことが確認できる

### 3. ネットワーク要件

目的: 管理画面への不正アクセスを防ぐ

要件:
- 管理画面へのアクセスは認証必須（無認証で公開しない）
- 通信は HTTPS のみ
- 一般的な Web 攻撃（SQLi・XSS・CSRF）への防御層を導入

受け入れ基準:
- アクセス制限ポリシーが定義されている
- Web 攻撃防御層の有無が明確

### 4. データ要件

目的: 個人情報を含むデータを安全に保管する

要件:
- データ所在地: 国内リージョン（日本国内）
- ユーザー情報: Cognito User Pool に保管。Cognito の標準動作に従う（保存時暗号化は Cognito 側で実施）
- 監査ログ: DynamoDB に保管。保管期間は1年以上、削除不可、保存時暗号化必須
- DynamoDB のバックアップを定期取得する（具体的な頻度・世代は基本設計で決定）

受け入れ基準:
- 保管期間ポリシーが文書化されている
- バックアップ取得運用が定義されている

### 5. ログ要件

目的: 不正検知・原因究明に必要なログを残す

要件:
- 管理者の管理操作: DynamoDB に Lambda から書き込む
  - **記録対象の操作**: ユーザー作成・編集・状態切り替え・パスワードリセット・一覧閲覧・監査ログ閲覧
  - 記録項目: タイムスタンプ・操作者ID・操作者名・操作種別・対象ユーザー・変更前後の差分・結果（成功/失敗）
  - 保管期間: 1年以上、追記専用、削除不可
  - ログに機密情報（パスワード・MFA 秘密鍵・セッショントークン）を含めない
  - **書き込み失敗時の対処**: Cognito 操作は成功したが DynamoDB への監査ログ書き込みに失敗した場合、CloudWatch Logs にエラーを出力し、CloudWatch Alarm で運用者に通知する。運用者は CloudTrail 等から実操作を確認のうえ、監査ログを手動で補完する（MVP では自動リトライは実装しない）
- 認証関連操作（ログイン・ログアウト・パスワード変更・MFA 操作）: CloudTrail に自動記録
  - 管理者・エンドユーザーともに対象
  - 独自の記録処理は実装しない
  - CloudTrail の保管設定は AWS の標準に従う

受け入れ基準:
- DynamoDB の記録対象操作リストが文書化されている
- 記録項目のスキーマが定義されている
- CloudTrail が有効化されている
- 監査ログ書き込み失敗時のアラート設定が文書化されている

### 6. 性能要件

目的: 管理者が業務を快適にこなせる応答速度を確保する

要件:
- 一覧表示・詳細表示は3秒以内
- 更新・削除等の操作は5秒以内
- メール送信などの通知は非同期で可

受け入れ基準:
- 主要操作の応答時間が目標値内

### 7. 可用性要件

目的: 管理者業務に支障が出ない範囲でシステムを利用可能とする

要件:
- 稼働時間: 平日業務時間に利用できればよい（24/7不要）
- 目標稼働率: 99.0%
- 計画停止: 業務時間外に実施可能
- 障害時の挙動: ID管理システムが停止してもユーザー側の Cognito ログインは継続できる

受け入れ基準:
- 月間稼働率が目標値以上

### 8. スケーラビリティ要件

目的: 想定範囲内のユーザー増加に対応できる構成にする

要件:
- 現在の規模: エンドユーザー数人〜数十人、管理者1〜数名
- 想定する将来規模: エンドユーザー数百人、管理者10名以下
- 上記範囲で構成変更を伴わずに対応可能

受け入れ基準:
- 想定規模で性能要件を満たす

### 9. 運用要件

目的: 自分1人で持続可能な運用体制とする

要件:
- 運用体制: 自分1人
- 運用時間: 必要時のみ（ベストエフォート）
- バックアップ取得状況の定期確認
- 主要ドキュメント整備: 構築手順・復旧手順
- ポートフォリオ閲覧者向けに、操作範囲を限定したテストユーザを1名事前作成する

受け入れ基準:
- 復旧手順書が文書化されている

### 10. コスト要件

目的: ポートフォリオ規模で持続可能なコスト水準を維持する

要件:
- 月額予算: 数千円程度
- コスト可視化: 月額の内訳を確認できる
- 予算超過アラート: 予算の80%・100%で通知発報

受け入れ基準:
- 月額が予算内
- 予算アラートが設定されている
